# v0.6.1 — Kilo-inspired Starter Pack + Hosted Catalog

> **Status:** design · **Author:** Claude · **Date:** 2026-06-13
> **Follows:** v0.6.0 Marketplace (M0–M4, shipped). **Origin:** Codex's "absorb Kilo's MCP/Skills" proposal.
> **Thesis:** the marketplace *mechanism* shipped in 0.6.0; this fills the *catalog* — and adds the vehicle (hosted fetch) so it can keep growing **without a new VSIX each time**.

---

## 1. Position (agree with Codex's direction, disagree on doing it all at launch)

Codex's analysis is right on substance: **adapt, don't bulk-copy.** Extract from Kilo's **public MIT repo** (not its online API), rewrite into Roam's own catalog entries, attribute properly. The reason this is 0.6.1 and not 0.6.0: every MCP entry must be **launch-verified or "Add" ships broken** (we hit `${ALLOWED_DIR}` literal + npx-vs-uvx bugs in M3 that a review had to catch). A large unverified catalog erodes trust faster than a small verified one. So 0.6.0 shipped the verified core (7 agents / 8 popular MCP); 0.6.1 expands it deliberately.

Note: **our 0.6.0 MCP set already IS the popular Kilo/Roo set** (filesystem, git, github, fetch, memory, sqlite, puppeteer, brave-search). The expansion is the long tail.

## 2. The three tiers (per Codex, confirmed)

| Content | Map to | Confidence |
|---|---|---|
| Kilo/Roo MCP servers | `marketplace/mcp.json` entries | **High** — direct, fastest. Each: `source` URL + `requiresApproval: true` for anything touching fs/net/creds |
| Kilo **modes** (Architect, Debugger, Ask, Orchestrator, custom) | `marketplace/agents.json` **Agent presets** — NOT skills | **High** — modes are role+model+prompt, which is exactly `AgentCatalogEntry`. `skills[]` MUST be real `SKILL_LIBRARY` ids (the M0 backstop enforces it) |
| Kilo `.kilocode/skills` | `marketplace/skills.json`, **rewritten** to Roam SKILL.md style | **Cautious** — browse/install draft only; don't promise every Kilo skill runs until the Skills loader (Phase 3) lands |

## 3. Hard requirements (the bug-prevention gate)

These are non-negotiable for any catalog expansion — they encode the M3 lessons:
1. **Per-entry MCP launch verification.** Before an MCP entry ships, actually launch it (`npx`/`uvx` resolves, server starts) — fold this into the bundle smoke or a dedicated `verify:mcp` step. No "assume npx works."
2. **Placeholders only `${WORKDIR}` (args) or secret `${VAR}` (env).** Anything else launches literal. (Runtime: `resolveServerPlaceholders` substitutes `${WORKDIR}`; `env` `${VAR}` resolved from SecretStorage.)
3. **Agent `skills[]` ⊆ SKILL_LIBRARY.** The M0 backstop fails CI otherwise — keep it; it's why M3's invented ids got caught.
4. **Licensing.** Only MIT repo content. Add `THIRD_PARTY_NOTICES.md` with Kilo's copyright + MIT text. **Never** label UI "Kilo marketplace" — use "Popular MCP Servers" / "Starter Pack"; provenance lives in each entry's `source` + NOTICES.
5. **Security-conservative.** All high-privilege servers default `requiresApproval: true` (already the norm).

## 4. The real expansion vehicle: hosted catalog fetch

In-repo JSON means every catalog addition = a VSIX release. That doesn't scale to an "ever-growing starter pack." The fix (design open-Q #4, now promoted):

```
loadCatalog already takes a reader; mergeCatalogs(base, override) already exists (M0 seam).
v0.6.1: at startup, fetch a Roam-hosted catalog JSON (best-effort, cached, short timeout) and
        mergeCatalogs(bundledCatalog, fetchedCatalog) — fetched wins on id collisions.
        Offline / fetch-fail → silently fall back to the bundled set (never blocks the panel).
```

- **Config:** `roam.marketplace.fetchCatalog: boolean` (default true), `roam.marketplace.catalogUrl: string` (Roam-hosted default).
- **Trust:** the fetched catalog is curated by Roam (same bar as in-repo), validated through the same `parse*` + backstop before merge; an invalid fetched section is dropped, not merged.
- **Payoff:** the Kilo-inspired long tail (and future community submissions) grows server-side; users get new agents/MCP/skills **without updating the extension**. This is also what lets the "Starter Pack" be 30+ entries without bloating the VSIX.

## 5. Phasing

1. **0.6.1a — hosted fetch** (enabler, code): `mergeCatalogs(bundled, fetched)` in the panel + the extension install handler, config flags, resilient fallback, tests. Small, unblocks everything else.
2. **0.6.1b — MCP starter pack**: +15–20 verified servers (postgres, redis, slack, gitlab, notion, google-maps, sentry, …), each launch-verified, sourced, approval-gated. Authored against the hosted catalog so it ships without a VSIX bump.
3. **0.6.1c — Agent presets from modes**: +Debugger, +Docs Writer, +Orchestrator-style, etc. — `AgentCatalogEntry` with real `SKILL_LIBRARY` ids.
4. **Skills** stay deferred until the Phase-3 on-demand loader; then expand `skills.json` (Roam-rewritten, attributed).

## 6. Open questions
1. **Who hosts the catalog JSON, and how is it updated?** A static file in a Roam-owned repo/CDN (cheapest, versioned) vs a service. Lean: static JSON on a Roam URL first; service only if community submissions arrive.
2. **Verification automation for MCP launch** — full launch (downloads packages, slow) vs a lighter "package resolves" check. Lean: light resolve-check in CI + a manual full-launch pass for new entries.
3. **Routing for authoring 0.6.1b/c** — same M3 pattern (RoamClaw authors against the schema + backstop, Claude verifies/launch-checks + merges). The backstop + a new `verify:mcp` make this safe to delegate.
