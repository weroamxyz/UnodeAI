# v0.7.0 Bug Audit — System Architect Review

## 1. Introduction

This document is the **read-only** bug audit of the v0.7.0 features in the UnodeAi codebase, performed by the System Architect agent. **No source files were modified** during this audit; the findings below are intended to inform the v0.7.0 release readiness decision and the post-release backlog.

The audit focused on the three v0.7.0 feature areas called out in the task brief:

- **Worktree concurrency strategy** — `src/backend/WorktreeCoordinator.ts`, `src/backend/FileCoordinator.ts`, `src/extension.ts` (worktree wiring)
- **Verification gate / merge gate** — `src/workflow/WorkflowEngine.ts`, `src/backend/TeamTools.ts` (`run_checks`), `src/extension.ts` (`runVerifyChecks`)
- **Settings / schema for worktree** — `src/settings/SettingsBridge.ts`, `src/types.ts`, `package.json` (`contributes.configuration`)

Across these surfaces, **62 distinct issues** were identified, classified by severity and by category (race conditions, missing error handling, incorrect git/worktree assumptions, gate leaks, type-safety issues).

> **Note on file naming:** The task brief referenced `src/backend/WorktreeFileCoordinator.ts`. That file does not exist; the implementation lives at `src/backend/WorktreeCoordinator.ts`. This audit treats that file as the canonical worktree-coordination source.

## 3. Critical Blockers (should block v0.7.0 release)

The three **high-severity** bugs are the only ones that can cause **silent data loss** or **silent policy bypass** in the new v0.7.0 worktree/verifier machinery. They should be fixed (or explicitly accepted as known-issues in the release notes) before shipping v0.7.0.

### Blockers

- **Bug #1 — Branch leak after worktree remove.** A user who runs a team, removes an agent, and re-creates an agent with the same `id` (the common edit-then-restart path) will hit `fatal: A branch named 'roam/<role>-<short>' already exists.` The agent silently falls back to the shared workspace, the user doesn't see why, and the worktree strategy degrades. The fix is small: either delete the branch in `WorktreeManager.remove` (best-effort `git branch -D`), or switch the `git worktree add` call to use `-B` (force create-or-reuse).
- **Bug #2 — `release()` vs in-flight `mergeAgent` race.** A completed-but-not-yet-merged turn combined with a user-driven "remove agent" call can result in the agent's work being committed to a branch whose worktree has already been deleted. The error is swallowed by the serialized chain's `.then(_, _)`. The user sees no failure, the agent's work is gone. The fix is to await the in-flight merge for the released agent before deleting the worktree.
- **Bug #3 — `runVerifyChecks` uses `cp.exec` with `timeout`.** On Windows, `cp.exec` with a timeout does NOT kill the child process (no SIGTERM semantics); the process keeps running, the gate returns a confusing `TimeoutError` with no output, and the workflow is misclassified. A watch-mode or stdin-waiting test stalls the whole crew for the full 10-minute window. The fix is to reuse the `cp.spawn` + `setTimeout` + `proc.kill('SIGKILL')` pattern from `verifyCommandRunner` (or factor it out to a shared helper).

### Release-readiness recommendation

**Do not ship v0.7.0 to end users until at least Bugs #1 and #2 are fixed.** Both are silent-data-loss bugs in the headline feature of this release (per-agent worktrees). Bug #3 should be fixed in the same release cycle because the verifier-as-gate is the second headline feature, and a Windows-breaking timeout is a regression from v0.6.x (where the verifier didn't exist).

## 5. Low-Severity Findings (omitted)

In addition to the 20 bugs above, the full audit identified **42+ additional low-severity findings** across the same v0.7.0 surfaces. These include:

- String-based error detection in `awaitTasks` (prefix matching for `[BLOCKED`)
- Inconsistent output caps between `Verifier` (`8000` chars) and `runVerifyChecks` (`4000` chars)
- `tierController?.applyTiers(...)` not `await`ed in `WorkflowEngine.handleGate`
- `Worktree.friendlyName` uses `Math.random()` (cosmetic; collision probability is negligible at the default cap of 4)
- `mergeChain.then(fn, fn)` pattern is non-idiomatic; works but interleaves log output
- `WorktreeCoordinator.notifyAgent` sends to a non-existent session after `release()`
- `SettingsBridge.setApiKey` accepts whitespace-only keys
- `gatherWorktreeReview` does not differentiate lanes in the `verifying` state
- `WorkflowEngine.restore` silently skips instances whose step's role is gone
- `commitWorktree` runs as `UnodeAi <roam-crew@example.invalid>` — fine for local commits but flagged as suspicious by some git hosting providers
- `WorktreePanel` re-renders on every refresh; no debounce

These are out of scope for the release decision and are tracked separately in the post-release backlog.

## 7. Verification verdict (Claude — gate review, 2026-06-15)

Each finding was verified against the actual code (not taken on faith). Result: **6 real → fixed this release**, 5 not-real/already-correct, the rest deferred to backlog.

### Fixed in this release
- **#1 Branch leak** — real (low-likelihood but cheap). `WorktreeManager.remove` now takes `pruneBranch`; `WorktreeCoordinator.release` passes it so the agent's `roam/<name>` branch is deleted after its work is on integration. Test: real-git "re-create same name succeeds".
- **#2 release() vs in-flight merge** — real race. `release()` is now serialized on the same `mergeChain`, so an in-flight commit/merge for the agent finishes before its worktree is removed. Test: ordering proof (`['merge','remove']`).
- **#3 `runVerifyChecks` `cp.exec` timeout** — real (Windows). Now reuses `verifyCommandRunner` (`cp.spawn` + timeout + `SIGKILL`), the same robust path as the worktree gate.
- **#5 `run_checks` no timeout** — real. `defaultRunner` now has a 300s wall-clock cap that kills the child.
- **#11 discarded merge promise** — real (hygiene). `onTurnComplete` is now `.catch`-logged.
- **#20 schema missing `minimum`** — real. `maxParallel` → `minimum:1`, `verifyTimeoutSeconds` → `minimum:10`.

### Not real / already correct (no change)
- **#10 & #15 — `ask`/`none` "silently bypassed" by `run_checks`/`runVerifyChecks`.** Incorrect: both **block** on `!verdict.allowed` (which includes `ask`). They are stricter than claimed, not laxer. (The actual `ask`-bypass was in `Verifier`, already fixed pre-audit in the Codex review.)
- **#8 — auto-finalize "not serialized".** Incorrect: both `finalize()` and the auto-finalize inside `mergeAgent` run on the single `mergeChain`; calling `runFinalize` directly inside `mergeAgent` is deliberate (nesting `serialize` would deadlock). They cannot run concurrently.
- **#4 — stale `commandPolicy`.** Incorrect: `commandPolicy` is mutated in place via `.reload()` on config change and every consumer holds the same reference (the verify thunk reads it live), so it is never stale.
- **#6 — `run_checks` uses PM's `workingDirectory`.** Not an issue in practice: the PM is non-isolated, so its cwd resolves to `workspaceRoot()`.

### Deferred to backlog (real-but-low-value or rare edge cases)
#7 (soft cap race — at most one extra worktree), #9 (`update-ref` with base checked out elsewhere — rare; `update-ref` doesn't refuse), #12 (detached HEAD), #13 (`--no-ff` finalize merge — works; fix the comment), #14 (workflow `system.error` no retry — spec decision), #16 (adopted-worktree clean check), #17 (user on `roam/integration` guard), #18 (review-panel mid-merge snapshot — cosmetic), #19 (`add -A` scope), plus the §5 low-severity list. None block the release.
