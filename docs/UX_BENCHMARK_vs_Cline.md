# UX Benchmark — UnodeAi vs Cline(同模型对标)

> **状态**:现行评测协议(Claude 维护)· **日期**:2026-06-10 · **基线**:UnodeAi v0.5.1
> **目的**:让 Cline 和 UnodeAi **跑同一个模型**,在同一组真实任务上并排比体验,**找出 UnodeAi 体验落后
> Cline 的地方**,并要求通过 **V0.5.x 小版本迭代**逐项补齐,**不允许 UnodeAi 的用户体验落后于 Cline**。
>
> **为什么必须同模型**:UnodeAi 的卖点是「便宜模型套利 + 多 agent」。如果用便宜模型时体验比 Cline(配强
> 模型)差,用户会把「便宜模型不好用」错怪到产品上。所以对标必须**控制变量 = 同一个模型**,隔离出「纯产品/
> 交互体验」的差距,这部分是我们能且必须补的。护城河功能(多 agent/团队/成本)走 V0.6.0,**不在本对标范围**。

---

## 一、方法论(控制变量)

| 维度 | 设定 |
|------|------|
| **模型** | 两边配**完全相同**的模型与参数。基准跑 **3 档**:① 便宜档(如 deepseek/qwen-flash,主场景)② 中档(如 deepseek-v3/kimi)③ 强档(如 claude/gpt,作为「体验天花板」参照)。每档两边一致。 |
| **任务** | 固定一组 8 个代表性任务(§三),两边跑同一任务、同一 prompt、同一仓库初始状态。 |
| **环境** | 同一台机、同一 VS Code、同一目标仓库(每次任务前 `git stash`/重置到干净基线)。 |
| **UnodeAi 模式** | 默认用 **Solo/Fast 模式**对标(Cline 是单 agent;同构才公平)。Team 模式是 UnodeAi 独有优势,**不在本对标**(那是 V0.6.0 护城河线的事)。 |
| **评分** | 每个任务按 §二 的 10 个体验维度打分(0–3),记录**具体差距证据**(截图/录屏/复现步骤),不止打分。 |
| **判定** | 任一维度 UnodeAi < Cline 即记为「**parity gap**」,进 §五 的 V0.5.x 迭代清单,标优先级。 |

> 评测者:Claude 设计协议 + 汇总;DeepSeek/你实跑并录证据;Claude 把 gap 转成任务卡。

## 二、体验维度(10 项 scorecard,0–3 分)

> 0 = 缺失/糟糕 · 1 = 能用但明显差 · 2 = 接近 Cline · 3 = 持平或更好。**目标:每项 ≥ Cline。**

| # | 维度 | 看什么(UnodeAi 现状锚点) | 已有能力 |
|---|------|------------------------------|----------|
| U1 | **流式手感** | 首 token 延迟、token 流畅度、不卡顿不闪烁、滚动不打架 | v0.5.1 增量渲染已上 |
| U2 | **Plan/Act 清晰度** | 是否先给计划再动手、计划可见可改、实时 Todo 勾选 | 实时 Todo(C3)+ Plan/Act 模式已有 |
| U3 | **Diff / 写文件审批** | 改动以 diff 呈现、审批/批量审批/拒绝手感、改后能不能看清改了啥 | 写审批(V2,diff 预览)已有 |
| U4 | **回退 / checkpoint** | 一键撤销这一轮、恢复某个文件到改前、跨 reload 存活 | Checkpoints/Restore(V1)已有 |
| U5 | **错误恢复手感** | 工具失败/命令报错时,agent 自己读错误并修;不死循环、不甩锅「环境坏了」 | 弱模型鲁棒四层 + 命令改写已有 |
| U6 | **上下文附加(@)** | @file/@folder/@problems/@url 的发现性、补全、塞入是否顺滑 | @-context 全套已有 |
| U7 | **终端可见性** | 命令在真终端跑、看得到输出、能介入 | 每 agent 真终端(PTY)已有 |
| U8 | **运行中介入(插话)** | 跑到一半能不能插话纠偏 / 排队下一条 / 打断 | ⚠ 现在运行时输入框禁用(BACKLOG #6)→ **已确认 gap,立项 G-001 / V0.5.2(见 §五)** |
| U9 | **成本/用量透明** | 每轮 token/成本、context %、能不能心里有数 | 实时 metrics + 成本路由已有 |
| U10 | **上手/空状态** | 第一次打开到跑通第一个任务的顺滑度、onboarding、空状态引导 | onboarding 两门 + Solo 默认已有 |

> 备注:多数维度 UnodeAi **已有对应能力**,本对标是去验「手感是否真的 ≥ Cline」,而不是从零造。U8(运行中介入)
> 是已知的结构性 gap(输入框运行时禁用),很可能是第一个要补的。
