## v0.2.1 hotfix candidates found after v0.2.0 release

### Startup restores the wrong team instead of the last working state

**Symptom**: after restarting VS Code, UnodeAi can return to an older team state instead of the final
team the user was working with. In the current workspace, `.roam/team.json` contains a single `Browser`
agent with a Playwright MCP server, so startup can appear to "default to browser agent".

**Confirmed cause**:
- Startup restored `.roam/team.json` members whenever the file existed, even when workspaceState had the
  user's last runtime roster. That makes an old or validation-only `team.json` shadow the real last-used
  team.

**Hotfix status**:
- Local fix prepared: startup now restores the last workspace roster first; `.roam/team.json` members are
  used only as a seed when no runtime roster exists. The MCP server registry still loads from
  `.roam/team.json`.
- Startup wizard now opens only when no agents are restored.

### Model Tuning save closes or resets the Settings panel

**Symptom**: when tuning models for multiple agents, pressing Save for one agent refreshes the Settings
panel, forcing the user to reopen or re-navigate before editing the next agent.

**Confirmed cause**:
- The Settings panel re-rendered the entire webview after every message, including `saveTuning`.

**Hotfix status**:
- Local fix prepared: `saveTuning` persists the selected agent's parameters without re-rendering the
  panel. A separate Close button now owns the close action.

### Model picker shows only 5 models when live catalog is unavailable

**Symptom**: the Add/Edit Agent model picker can suddenly show only the 5 built-in static Roam models
instead of the full live gateway model list.
Users may also experience failed sends if `ROAM_API_KEY` is absent or the configured base URL is blank or
incorrect.

**Confirmed causes**:
- The picker falls back to static `DEFAULT_PROVIDER_CONFIGS.roam.models` when `ModelCatalog.list()` cannot
  fetch live `/models`; that static list currently has 5 models.
- A blank `roam.baseUrl` setting can skip the live endpoint in the dialog path unless the picker treats
  blank as "use the default Roam gateway".
- The default gateway `https://www.unodetech.xyz/v1/models` returns 401 without `ROAM_API_KEY`; if the key
  is missing from VS Code SecretStorage, the picker also degrades to static models.

**Hotfix status**:
- Local fix prepared: normalize the picker base URL with `resolveModelCatalogBaseUrl()` so blank Roam URLs
  fall back to `https://www.unodetech.xyz/v1`.
- Regression test added for blank Roam base URL fallback.
- Verified locally: `compile`, `lint`, full unit suite, and E2E all pass.
- Not committed in this environment because `.git/index.lock` cannot be created (`Permission denied`).

**Follow-up UX improvement**: when live `/models` fails, show a non-blocking QuickPick item or warning such
as "Live model list unavailable; showing built-in defaults. Check ROAM_API_KEY / base URL." Today the
fallback is silent, which makes the 5-model list look like a product regression instead of a connectivity
or credential fallback.

## 1. Context compaction by summarization (the proper fix) 🔴 high value

**Why**: v0.1.1 ships anchor-aware *truncation* (keep system + first user task + recent tail, drop the
middle, down to the ~70% soft budget) plus an 80% hard gate. That prevents the degradation band but
still **loses information** from the dropped middle on long sessions — which surfaces as the agent
"forgetting" earlier decisions (looks like hallucination).

**Design**:
- Trigger at the **soft threshold (70%)** — proactive, leaves headroom before the 80% hard brake.
- Replace the dropped middle turns with an **LLM-generated running summary** instead of discarding them:
  keep `[system (+project_context)] → [rolling summary of older turns] → [anchor: first user task] →
  [last K full turns]`.
- Generate the summary with an **economy-tier model** (cheap), reusing the tier infra; cache it and
  extend it incrementally (summarize only the newly-dropped turns, append to the prior summary).
- **Scope: OpenAI-compatible backend only.** The Claude headless backend has native compaction — do
  NOT double-compact it; let `claude` manage its own window.
- Token-aware throughout (build on `TokenCounter.softLimit()` and `estimateMessages()`).

**Touch points**: `OpenAICompatBackend` (replace/augment `trimHistory` with a summarize-then-trim path),
a small summarizer helper (its own cheap chat call), `TokenCounter` (already exposes soft/hard).

**Acceptance**: a long session stays under the window while a probe for an early decision (e.g. the
architect's contract) is still answerable from the summary; summary cost is bounded (incremental).

## 2. PM → Claude backend delegation 🔴 differentiation

Today PM delegation (`TeamTools`) only drives in-process (openai-compat) agents. Host the existing
`TeamMcpBridge` (core done) as a local MCP endpoint (streamable-http or stdio) and wire `claude
--mcp-config` so a Claude-backed PM can orchestrate too. Needs an IPC/process epic + live verification.

### Deferred chat idea — multi-tab "named dialog threads" (post-C4, evaluate for v0.3)
A DeepSeek draft proposed replacing the single sidebar chat with a **multi-tab DialogManager**: N open
tabs, each a *named, persisted* conversation thread with its own Plan/Act mode and transcript (Cline-style
task list). **Not adopted for v0.2.0** — it would discard the just-merged C1/C2 (single `roam.chat`
WebviewView + agent switcher, chosen deliberately) and balloon C3/C4 scope. The Chat-Parity plan stays
C1→C4 (one sidebar view, per-agent switcher, streaming, tool cards, Plan/Act). Revisit "named multi-thread
dialogs" as a separate v0.3 enhancement on top of the C1–C4 base if users want parallel named threads.

### Streaming for thinking models + tool calls (found in E3 live validation) 🟡
Observed during E3.1 with `deepseek-v4-pro` on the unodetech gateway: the **streamed** tool-calling turn
fails with `HTTP 400: "reasoning_content in the thinking mode must be passed back to the API."`, then
`OpenAICompatBackend` **gracefully falls back to non-streaming** and the turn completes correctly. So results
are right, but thinking models lose live token streaming in chat. Fix options: echo the prior assistant
turn's `reasoning_content` back on streamed follow-ups, or detect thinking models and skip streaming for
them. Non-blocking (correctness is fine via fallback). See [MCP_LIVE_VALIDATION_REPORT.md](MCP_LIVE_VALIDATION_REPORT.md).
