## 1. Introduction

This is a **read-only** re-evaluation of the v0.7.0 bug audit originally recorded in `docs/v0.7.0_BUG_AUDIT.md`. The same source files were re-read and the original top-20 findings were re-assessed. **No source files were modified.**

## 3. New Findings (Kimi perspective)

| # | File | Severity | One-line description |
|---|------|----------|----------------------|
| N1 | `src/backend/WorktreeCoordinator.ts` → `WorktreeManager.ts` | **high** | `doRelease` passes `{ force: true, pruneBranch: true }` to `manager.remove`, but `remove` only accepts `{ force?: boolean }`. The branch is **not pruned**, so Bug #1 is un-fixed and the code has a TypeScript excess-property error. |
| N2 | `src/backend/TeamTools.ts` + `src/extension.ts` (`defaultRunner` / `verifyCommandRunner`) | **high** | Both runners use `spawn(command, { shell: true, ... })` + `proc.kill('SIGKILL')`. On Windows, killing the shell (`cmd.exe`) does **not** kill grandchild processes spawned by the shell. A watch-mode test or input-waiting command can leave orphan processes. |
| N3 | `src/extension.ts` (`runVerifyChecks`) + `src/backend/TeamTools.ts` (`runChecks`) | medium | Inconsistent handling of `commandApproval: 'ask'`: `Verifier.verify` (worktree) now skips, but `runVerifyChecks` (workflow gate) and `TeamTools.runChecks` (PM) still run without prompting. |
| N4 | `src/extension.ts` (`runVerifyChecks`) + `src/backend/TeamTools.ts` (`runChecks`) | medium | `commandApproval: 'none'` is documented as "no shell commands at all", but both `runVerifyChecks` and `runChecks` still execute the verify command. |
| N5 | `package.json` + `src/extension.ts` (`verifyCommandRunner`) | low | `roam.worktree.verifyTimeoutSeconds` has `minimum: 10` but no `maximum`. A user setting a huge value (e.g., `999999`) creates an effectively unbounded gate stall. |
| N6 | `src/backend/WorktreeCoordinator.ts` (`release`) | low | `release()` serializes on the global `mergeChain`, so removing agent B waits for an in-flight merge of agent A. Correct but over-serialized; a per-agent queue would be more responsive. |
| N7 | `src/extension.ts` (`changedFilesInWorktree` / `gatherWorktreeReview`) | low | Both helpers use `runGitInRoot(['rev-parse', '--abbrev-ref', 'HEAD'])` and fall back to `'HEAD'` literal on detached HEAD — same bug as #12, affecting anti-cheat diff base and review board. |
| N8 | `src/extension.ts` (`runVerifyChecks`) + `src/backend/Verifier.ts` | low | Output cap inconsistency: `runVerifyChecks` slices to 4000 chars, `Verifier` caps at 8000. |
| N9 | `src/extension.ts` + `src/backend/TeamTools.ts` | low | `verifyCommandRunner` and `defaultRunner` duplicate nearly identical timeout/spawn logic. A shared `runCommandWithTimeout` helper would reduce divergence and future fix drift. |

### Why N1 and N2 are high

- **N1** means the headline "worktree remove" fix is incomplete. The next agent with the same `id` will still hit `fatal: A branch named 'roam/<name>' already exists.`, and the TypeScript build may fail (or silently ignore the option, depending on strictness). Either outcome blocks a clean v0.7.0 release.
- **N2** is the residual Windows risk after Bug #3's fix. The move from `cp.exec` to `cp.spawn` + `SIGKILL` is correct for Unix, but on Windows the signal terminates `cmd.exe` while leaving the actual test runner alive. Over a long session these orphans accumulate. The fix is to avoid `shell: true` (parse the command into args) or use `taskkill /T /F` on Windows.

## 4. Critical Blockers for v0.7.0

The following issues should block v0.7.0 release:

| # | Issue | Why it blocks |
|---|-------|---------------|
| **#1 / N1** | Branch leak + incomplete fix | The worktree remove path was changed to pass an unsupported `pruneBranch` option. The branch is not deleted, the same-id re-create fails, and the code may not compile. |
| **N2** | Windows orphan processes after verify timeout | The new `verifyCommandRunner` and `defaultRunner` still leave zombie processes on Windows because `shell: true` + `SIGKILL` kills the shell, not its children. |
| **#2** | Release vs in-flight merge race | Now fixed; no longer a blocker. |
| **#3** | `runVerifyChecks` `cp.exec` timeout | Now fixed; no longer a blocker. |

### Release-readiness recommendation

**v0.7.0 should still be blocked** until:

1. **N1 is resolved.** Either implement `pruneBranch` in `WorktreeManager.remove` (delete the branch after `worktree remove`) or revert to passing only `{ force: true }` and fix the leak differently (e.g., use `-B` in `create`). Do not ship with a half-applied fix.
2. **N2 is addressed.** On Windows, the verify/run_checks timeout kill must terminate the whole process tree. Options:
   - Avoid `shell: true` and split the command into executable + args.
   - Use `taskkill /T /F /PID <pid>` on Windows inside the timeout handler.
   - Share one robust `runCommandWithTimeout` helper for both runners.

The following are **not blockers** but should be in the first patch (0.7.0.x or 0.7.1):

- Bug #4 (stale `commandPolicy`)
- Bug #6 (`runChecks` cwd)
- Bug #8 (auto-finalize serialization)
- Bug #10 / N3 (inconsistent `'ask'` handling)
- Bug #15 / N4 (`'none'` mode bypass)
- Bug #9 (update-ref on branch checked out elsewhere)

## 6. Final Recommendation

**Do not release v0.7.0 yet.** Two of the three original high-severity blockers have been fixed, but the codebase has gained a **new high-severity issue (N1)** from an incomplete branch-leak fix, and a **residual high-severity Windows issue (N2)** from the verify-runner refactor. Both affect the headline v0.7.0 features (worktree isolation and verifier-as-gate) and both can produce user-visible failure or resource leaks.

Once **N1** and **N2** are fixed, the remaining medium/low items can ship in a follow-up patch. The core architecture is sound; the blockers are implementation gaps, not design flaws.

## 7. Claude verification verdict (gate review, 2026-06-15)

Verified each claim against the code at HEAD (this review was written against a pre-fix snapshot).

**Already fixed (Kimi read stale code):**
- **N1 / #1** — *not a current bug.* `WorktreeManager.remove` already accepts `{ force, pruneBranch }` and deletes the branch (commit `0b36fc7`); `tsc` is clean (no excess-property error) and the real-git test "pruneBranch deletes the branch → same-name re-create succeeds" passes. The branch IS pruned.
- **#2, #3, #5, #20** — confirmed fixed (matches Kimi).

**Real new finding → fixed this commit:**
- **N2** — *real.* The timeout did `proc.kill('SIGKILL')` on a `shell:true` spawn → on Windows kills `cmd.exe` but not its children. Added `killProcessTree` (Windows `taskkill /T /F`, POSIX SIGKILL), used by both `verifyCommandRunner` and `defaultRunner` (also dedups the kill path per **N9**). Tested.
- **N5** — added `maximum: 3600` to `roam.worktree.verifyTimeoutSeconds`.

**Not real (re-confirmed against code):**
- **N3 / N4 (= #10 / #15)** — *wrong.* Both `TeamTools.runChecks` and `runVerifyChecks` already `return`/block on `!verdict.allowed` (which includes `ask`); neither auto-runs on `ask`/`none`. The downstream treatment differs (worktree gate *skips→proceeds*, workflow gate *blocks→fails*) but that's by design, not a bypass.
- **#4** — *not stale.* `commandPolicy` is mutated in place via `.reload()` on config change and the verify thunk reads the live module reference, so policy updates ARE seen.
- **#8** — *not a race.* Both `finalize()` and the auto-finalize inside `mergeAgent` run on the single `mergeChain`; they cannot interleave.
- **#6** — non-issue in practice (PM is non-isolated → cwd = `workspaceRoot()`).

**Deferred to backlog (real-but-low/rare, unchanged):** #7, #9 (partially addressed by the shared `killProcessTree`), #12/N7 (detached HEAD), #13, #14, #16, #17, #18, #19, N6 (over-serialized release — correct, just not maximally responsive), N8 (output-cap cosmetic). None block release.

**Net from Kimi's review:** 1 genuine new fix (N2) + 1 trivial (N5). Higher signal than the prior audit, but its two "blockers" were a stale read (N1) and a real-but-medium robustness gap (N2) — not release-stoppers given #1/#2/#3 were already fixed.
