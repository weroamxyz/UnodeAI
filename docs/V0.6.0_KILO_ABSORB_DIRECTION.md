**Status**: direction set 2026-06-12 · **Gate**: finish v0.5.x (context injection) first.
**Premise**: Cline parity is done (R2 gaps all closed). v0.6.0 stops *catching up* and starts *differentiating* — by absorbing what Kilo Code (and its parent Roo Code) do well, on top of UnodeAi's existing multi-agent + cost-arbitrage base.

> ⚠️ **Honesty note**: my (Claude's) knowledge of Kilo's *exact* implementation is partial. So step 1 is a **grounded research pass** (DeepSeek), not design-on-assumptions. No implementation cards for the big feature until it's designed and 张 has reviewed.

---

## Two thrusts

### Thrust A — MCP / Skills depth
UnodeAi already has MCP (injection-side + claude-native) and a Skills→capability model. Kilo/Roo are known for: a one-click **MCP marketplace/install**, **custom modes**, rules/memory. The goal isn't to copy — it's to find the *specific* places our MCP/Skills UX is thinner and close them.
- Needs: a gap analysis (what Kilo does that we don't, in MCP discovery/install and custom modes/skills), then incremental cards.

### Thrust B — Solo fan-out into parallel git-worktree agents ⭐ (the headline)
Today: Solo = one agent in the open workspace; Team = PM-led agents sharing one workspace. **New**: Solo can **fan out into N parallel agents, each in its own git worktree**, work independently, then merge back. This is the differentiating capability 张 wants from Kilo.
- Why worktrees: true file isolation — parallel agents can't clobber each other's edits (the FileCoordinator problem disappears when each agent has its own working tree).
- Needs: **a design pass** (architecture) before any implementation — this touches process/lifecycle, git, the session model, and UI.
