# v0.6.0 — Marketplace + Header Information Architecture

> **Status:** design · **Author:** Claude · **Date:** 2026-06-13
> **Drives:** Kilo-absorb rec #3 (MCP marketplace) — extended to **Agents + MCP + Skills**, and the header reorg that makes room for it.
> **Pairs with:** [V0.6.0_WORKTREE_FANOUT_DESIGN.md](V0.6.0_WORKTREE_FANOUT_DESIGN.md). Skill side ties to gap-analysis rec #1 (progressive skill loading).

---

## 1. The problem

The `roam.teamPanel` native toolbar currently crams every control into one undifferentiated `navigation` group (`package.json` `view/title`):

`addAgent · solo · createTeamPreset · editTeamRules · restoreCheckpoint · startAllAgents · stopAllAgents · openSettings · collapse/expand`

**Extension-level** (`openSettings`) sits next to **team-level** (`addAgent`, `editTeamRules`), and there is **no home for a Marketplace**. The mental model is muddy: a user can't tell what acts on *this crew* vs. *the whole plugin*.

## 2. The principle — sort every affordance by scope

| Scope | Affordances | Lives in |
|---|---|---|
| **Extension** (all teams, all projects) | **Marketplace** (Agents/MCP/Skills), **Settings**, Docs/Help, Provider/account | **Row 1** — right of the "UnodeAi" brand |
| **Team** (current crew/session) | Add Agent, Create/Switch Team, Team Rules, Start All / Stop All, Solo toggle, Restore Checkpoint | **Row 2** — team sub-header |
| **View** (display only) | Collapse / Expand | Native VS Code toolbar (the only thing left there) |

## 3. Header layout (decided: two webview rows)

```
┌─ TEAM ───────────────────────────────(native, collapse/expand only)─┐
│ ⦿ UnodeAi  v0.5.12                    🛒 Marketplace   ⚙ Settings  │  ← Row 1: EXTENSION level
│ 📋 Dev Crew ▾    ＋Agent  ⤳Switch  ⧉Rules   ▶All ■All   ⚡Solo       │  ← Row 2: TEAM level
├─────────────────────────────────────────────────────────────────────┤
│  📋 Product Manager        ● running                                 │
│  🛠 Developer               ○ idle                                    │
│  🔍 QA                      ○ idle                                    │
└─────────────────────────────────────────────────────────────────────┘
```

- **Row 1** (`.rc-titlebar`, existing): brand + version stay left; a right-aligned action group is added (`justify-content: space-between`, brand group + actions group). Buttons are HTML, dispatching via the existing `data-command` → `postMessage` path the panel already uses.
- **Row 2** (new `.rc-teambar`): the team-level controls move here out of the native toolbar. Team name doubles as the **Switch Team** dropdown (replaces the separate `createTeamPreset` icon affordance, though the command stays).
- **Native `view/title`**: stripped to `collapse/expand` only. Everything else removed from `package.json` `menus.view/title` and re-homed in the webview rows.

Per-agent icons (📋/🛠/🔍 …) are unchanged — those are **role identity**, neither extension nor team chrome.

## 4. The Marketplace

A new webview (`roam.marketplace`, opened from Row 1) with three tabs. **Browsed globally, applied at a scope chosen on install** — that's what keeps it coherent with the IA:

```
┌ Roam Marketplace ───────────────────────────────────┐
│  [ Agents ]   [ MCP ]   [ Skills ]          🔍 search │
├──────────────────────────────────────────────────────┤
│  📋 Product Manager   requirements → spec    [ Add ▾ ]│
│  🛠 Security Auditor   SAST + secret scan     [ Add ▾ ]│
│  🎨 UX Reviewer        a11y + design crit     [ Add ▾ ]│
└──────────────────────────────────────────────────────┘
        Add ▾  →  ⟮ to current team ⟯ ⟮ create new team with it ⟯
```

### Catalog source (decided: curated JSON index in-repo)
```
marketplace/
  agents.json   → [{ id, name, role, skills[], model, tier, icon, systemPrompt }]
  mcp.json      → [{ id, name, transport: 'stdio'|'http', cmd|url, env[], description }]
  skills.json   → [{ id, name, description, body|SKILL.md path, capabilities[] }]
```
Shipped with the extension; optionally re-fetched from a Roam-hosted copy for updates without a release. No backend service in v1 (defer the hosted registry until the UX is proven — gap-analysis rec #3).

### Apply-scope per tab (the important part)
| Tab | Browse scope | Install action | Applied scope |
|---|---|---|---|
| **Agents** | global catalog | "Add to current team" / "Create team with it" | **Team** — reuses `addAgent`; an Agent entry is just a `RoleConfig` (`src/roles/RoleConfig.ts`) authored as data |
| **MCP** | global catalog | generate config → **approval gate** | **Extension** (all agents) *or* per-team/agent; flows through the **existing MCP approval model** — marketplace just removes the hand-typed JSON |
| **Skills** | global catalog | install `SKILL.md` to `~/.roam/skills/<id>/` (global) or `.roam/skills/` (project) | attach to a role/agent; loaded **on-demand** (gap-analysis rec #1 progressive disclosure) |

So the Marketplace is the **discovery front-end** over two systems we're already planning: the **MCP registry** (rec #3) and the **on-demand Skills** loader (rec #1). It's not a fourth thing — it's the UI that makes those two browsable, plus Agents-as-data.

## 5. Code mapping

- `package.json` `menus.view/title` — remove all team/extension items except `collapse/expand`; keep the `roam.*` **commands** (they still exist, just invoked from the webview now).
- `src/views/TeamViewProvider.ts` — `.rc-titlebar` gains a right-aligned actions group (Marketplace, Settings); add `.rc-teambar` row with the team-level buttons; wire their `data-command` dispatch.
- New `src/views/MarketplacePanel.ts` — webview, 3 tabs, reads `marketplace/*.json`, renders cards, `Add ▾` scope picker → dispatches `addAgent` / MCP-config-generate / skill-install.
- `marketplace/*.json` + a tiny `MarketplaceCatalog` loader (validate shape, merge in-repo + fetched).
- Agents tab leans entirely on existing `RoleConfig`/`addAgent`; MCP tab on the existing MCP host + approval; Skills tab needs the on-demand loader (rec #1) — so **Skills tab ships when progressive skill loading does**, Agents + MCP tabs can ship earlier.

## 6. Phasing

1. **Header IA reorg** (small, ships first, independent): two-row layout, scope-sorted controls, strip native toolbar. Pure UX win, no marketplace dependency. Add a `🛒 Marketplace` button that opens a "coming soon" stub.
2. **Marketplace · Agents + MCP tabs**: curated JSON, scope-picker install. Agents reuse `RoleConfig`; MCP reuses approval gate. This is the rec-#3 leapfrog.
3. **Marketplace · Skills tab**: arrives with on-demand skill loading (rec #1) — the two are the same workstream.

## 7. Open questions

1. **Marketplace surface**: full webview panel (proposed) vs. a native QuickPick list? Webview is richer (cards, search, tabs) and matches Kilo's framing; QuickPick is cheaper but cramped for 3 categories. Lean webview.
2. **Switch-Team via brand dropdown**: does folding `createTeamPreset` into the Row-2 team-name dropdown lose discoverability? Keep the command in the palette regardless.
3. **MCP apply-scope default**: when a user adds an MCP server, default to extension-wide (all agents) or ask each time? Lean: ask once, remember per server.
4. **Hosted catalog updates**: ship purely in-repo (update only on extension release) or fetch a Roam-hosted JSON at startup? Lean: in-repo first, add fetch when the catalog grows.
