# v0.6.0 — Worktree Fan-Out Design (orchestrated parallel agents)

> **Status:** design · **Author:** Claude · **Date:** 2026-06-13
> **Unblocked by:** [KILO_GAP_ANALYSIS.md](KILO_GAP_ANALYSIS.md) §2 (corrected — Kilo's Agent Manager confirmed to do parallel worktrees).
> **Theme:** the v0.6.0 moat item. Absorb Kilo's worktree isolation; pass it with orchestration + an auto-merge loop Kilo deliberately omits.

---

## 1. Why — the corrected competitive frame

The gap analysis originally said Kilo was single-agent. **It isn't.** Kilo's **Agent Manager** runs parallel agents, each in its own git worktree under `.kilo/worktrees/`, with Multi-Version Mode (≤4 runs of the same prompt, model compare). So:

| | Kilo Agent Manager | Roo Worktrees | UnodeAi today | UnodeAi v0.6.0 |
|---|---|---|---|---|
| Per-agent filesystem isolation | ✅ worktree | ✅ worktree (per VS Code window) | ❌ same branch | ✅ worktree |
| Multiple agents in **one** session | ✅ (independent) | ❌ (window-level) | ✅ (shared branch) | ✅ (isolated) |
| Coordination (PM decomposes 1 goal across roles) | ❌ | ❌ | ✅ | ✅ |
| Shared team memory + inter-agent messaging | ❌ | ❌ | ✅ | ✅ |
| Merge-back | manual (Apply-to-local / PR) | manual (git) | n/a (same branch) | **auto-merge → PM review → main** |

**The single capability we are behind on is isolation.** Everything else is already ours. So v0.6.0 = add isolation **without** giving up coordination, and turn the manual merge step (Kilo's stopping point) into an automated PM-reviewed loop. That combination — *isolation + coordination + auto-merge* — is what neither Kilo nor Roo has.

Non-goal: matching Kilo's Multi-Version "race the same prompt N ways, human picks" UX. That's a model-comparison toy; our value is decomposed collaboration, not bake-offs. (Could be a later cheap add — see §7.)

---

## 2. Two workflows this serves

1. **Solo fan-out** (张's original ask: *"in Solo mode it can deploy more agents but in different worktrees like Kilo"*). A Solo agent that hits a task big enough to parallelize spawns N sub-agents, each in its own worktree, each on a slice. This is our answer to Kilo's Agent Manager — but the Solo agent (or PM) owns recomposition instead of the human.
2. **Team isolation.** An existing crew, but each role-agent works in its own worktree instead of stomping the shared branch. The PM already assigns and tracks; we swap the filesystem substrate underneath.

Both reduce to the same primitive: **per-agent worktree + a merge orchestrator.** Build the primitive once; expose it to both.

---

## 3. Architecture

The existing `FileCoordinator` (`src/backend/FileCoordinator.ts`) already declares `ConcurrencyStrategy = 'optimistic' | 'worktree'` and the extension point in `makeFileCoordinator()` (`src/extension.ts:818`) already branches on `roam.concurrencyStrategy === 'worktree'` and falls back with a warning. **But the FileCoordinator interface alone is insufficient** — it arbitrates read/write *hashes* on one shared tree. Worktrees change *where each agent's tree is*. So v0.6.0 introduces a sibling concept, not just a new coordinator impl:

```
WorktreeManager                 ← lifecycle: create / list / remove git worktrees
  .create(agentId) -> Worktree { path, branch }
  .remove(worktree, { force })

AgentWorkspace (per agent)      ← each agent's fs root is its worktree path, not the repo root
  - tool file ops (read/write/run_command cwd) resolve against worktree.path
  - injected where SessionManager builds the per-agent runtime (extension.ts:~206, the isSolo block)

MergeOrchestrator               ← the differentiator. On agent attempt_completion:
  1. commit the worktree branch
  2. attempt auto-merge into the integration branch (octopus-safe: serialize merges)
  3. clean merge  -> notify PM (send_message) for review gate
     conflict     -> hand the conflicting hunks back to the agent (or PM) to resolve, retry
  4. PM approves -> fast-forward/merge integration branch -> main; remove worktree
```

`WorktreeFileCoordinator` still exists but becomes thin: within a worktree an agent can't clobber a teammate (separate trees), so its `checkWrite` is effectively `NoopFileCoordinator` **until merge** — the real conflict surface moves from write-time to merge-time. That's the key architectural shift: **optimistic CAS catches conflicts per-write; worktrees defer them to a single merge gate.** Cross-agent staleness (the `takeStaleNotices` mechanism) is replaced by merge-conflict feedback.

### Where it plugs into existing code
- `makeFileCoordinator()` (`extension.ts:818`) — replace the warning stub with real wiring.
- The `isSolo` runtime block (`extension.ts:206–211`) already special-cases solo and swaps the coordinator — this is exactly where per-agent `cwd`/root redirection goes.
- `SessionManager` dispatch — on `attempt_completion`, route through `MergeOrchestrator` instead of straight to done.
- PM notification reuses the **existing** `messageBus.send(..., pm.config.id, 'ask.question', ...)` path already added for roster-change notify (0.5.12) — the review gate is the same primitive.
- Checkpoints: worktree branches make rollback trivial (drop the branch) — complements, doesn't replace, the planned shadow-git checkpoints (gap-analysis rec, §4 Roo).

## 4. Lifecycle (Team isolation path)

```
PM assigns task → spawn agent
  └─ WorktreeManager.create(agentId)         # git worktree add .roam/worktrees/<friendly-name> -b roam/<agent>/<task>
       └─ agent works (all file ops scoped to worktree path)
            └─ agent emits attempt_completion
                 └─ MergeOrchestrator: commit → merge into roam/integration
                      ├─ clean → PM review gate (diff shown via existing renderDiffDetail)
                      │           └─ approve → merge integration → main, remove worktree
                      │           └─ reject  → reopen task in same worktree with PM feedback
                      └─ conflict → feedback to agent: "your branch conflicts with <teammate> on <files>, rebase+resolve"
```

Worktrees live under `.roam/worktrees/` (mirror Kilo's `.kilo/worktrees/`), added to `.git/info/exclude` (local-only, no commit needed — Kilo's trick). A `.worktreeinclude`-style copy for untracked deps (`node_modules`, `.env`) — defer to phase 2; on Windows, copying `node_modules` is slow, prefer symlink/junction or shared install.

## 5. Windows / edge concerns (we ship Windows-first)

- **Path length**: nested `.roam/worktrees/<name>/...` can blow the 260-char limit. Use short friendly names (Kilo's "sunny-cloud" style) and document `core.longpaths=true`.
- **`index.lock` races on concurrent `git worktree add`**: Kilo serializes git-writes per repo root via a lock map. Mirror this — a `Mutex` around `WorktreeManager.create`.
- **Dirty main tree**: refuse fan-out if the user's working tree is dirty (or stash/warn) — worktrees branch from HEAD; uncommitted changes won't propagate.
- **Non-git workspace**: feature requires a git repo. Gate the UI; offer `git init` (consistent with how we gate other git features).
- **File watchers / TS server**: N worktrees = N copies the language server may index. Keep `.roam/worktrees/` out of the VS Code `files.watcherExclude` by default to avoid editor churn.

---

## 6. Config & UX surface

- `roam.concurrencyStrategy: "optimistic" | "worktree"` (already exists) — promotes worktree from stub to real.
- `roam.worktree.autoMerge: boolean` (default true) — false = stop at "ready to merge", user/PM merges manually (the Kilo-parity fallback).
- `roam.worktree.maxParallel: number` (default 4) — matches Kilo's ceiling; bounds resource use.
- Solo fan-out trigger: explicit (a `fan_out` tool / "split this" affordance), **not** automatic in v0.6.0 — auto-decomposition is a quality risk; let the Solo agent or user opt in.

## 7. Phasing

**Phase 1 (v0.6.0 core) — make worktrees real for Teams.**
`WorktreeManager` + per-agent root redirection + `MergeOrchestrator` with PM review gate + auto-merge of clean merges + conflict-feedback-to-agent. Windows-safe (lock + short names + dirty-tree guard). This closes the *only* capability gap and ships the auto-merge differentiator.

**Phase 2 — Solo fan-out.**
`fan_out` affordance for the Solo agent: spawn N worktree sub-agents on slices, Solo recomposes. Reuses Phase 1 primitives.

**Phase 3 — reach / parity polish.**
`.worktreeinclude`-style untracked-file copy; per-worktree terminal scoping; optional Multi-Version mode (race-the-same-prompt model compare) as a cheap Kilo-parity extra; worktree-aware checkpoints.

## 8. Open questions (resolve before Phase 1 build)

1. **Integration branch vs direct-to-main merge?** Proposed: a `roam/integration` staging branch so multiple agents merge there first, PM reviews the *combined* result, then one merge to main. Avoids N noisy commits on main. Confirm this matches how teams want history to read.
2. **Conflict owner**: when two agents' worktrees conflict at merge, who resolves — the *second* agent (rebase-and-retry), or the PM? Lean: agent first (it has the most context on its own change), escalate to PM after one failed retry.
3. **Claude-headless agents**: the subprocess backend runs commands in a cwd — confirm we can point each subprocess at its worktree path cleanly (should be straightforward: spawn `cwd: worktree.path`).
4. **Memory/notes during isolation**: `.roam/memory/notes.md` lives in the repo. If each agent is in a worktree, do they share one notes file (via the main checkout, outside their worktree) or per-worktree copies that merge? Lean: shared single file at the *real* repo root, accessed out-of-worktree, so team memory stays coherent (this is our advantage over Kilo — don't fragment it).

## 9. The one-line pitch (for README / changelog when it ships)

> UnodeAi agents now work in **isolated git worktrees** — no more stepping on each other's edits — and the PM **auto-merges** each agent's branch back through a review gate. Kilo isolates agents; we isolate them *and* keep them coordinated.
